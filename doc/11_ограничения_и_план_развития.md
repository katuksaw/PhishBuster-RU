# PhishBuster RU — Ограничения проекта и план развития

## Часть A: Анализ ограничений

### 1. Ограничения Manifest V3

| Ограничение | Описание | Влияние на проект |
|---|---|---|
| Service Worker «засыпает» | Background script может быть завершён Chrome через 30 секунд неактивности | Необходимо перезагружать rules.json при «пробуждении» |
| Нет доступа к `chrome.webRequest.onBeforeRequest` | Manifest V3 ограничивает перехват сетевых запросов | Невозможно анализировать HTTP-заголовки запросов |
| Запрет на удалённый код | Нельзя загружать JS с внешних серверов | База правил должна быть локальной (rules.json) |
| Ограничение на `host_permissions` | Требуется `<all_urls>` для анализа всех сайтов | Пользователь может не дать разрешение |

**Митигация**:
- Service Worker: загружаем rules.json при каждом сообщении (не только при инициализации)
- Локальная база: обновление только через обновление расширения
- Разрешения: объясняем пользователю через описание в manifest.json

**Что сказать на защите**:
_«Manifest V3 накладывает определённые ограничения, но они являются осознанным компромиссом Google в пользу безопасности и производительности. Мы проектируем архитектуру с учётом этих ограничений, что демонстрирует инженерную зрелость подхода.»_

### 2. Отсутствие доступа к SSL-сертификатам

**Проблема**: Chrome Extension API не предоставляет доступ к информации о SSL/TLS-сертификате сайта. Невозможно проверить:
- Кто выдал сертификат
- Срок действия
- Соответствие домена сертификату

**Влияние**: Фишинговый сайт с валидным (но выданным злоумышленнику) сертификатом не будет обнаружен по этому признаку.

**Альтернатива, которую можно использовать**: Проверка протокола (`https://` vs `http://`) — это доступно через `window.location.protocol`.

**Митигация**: Сосредоточиться на других метриках (домен, селекторы, ключевые слова), которые не зависят от SSL.

**Что сказать на защите**:
_«Отсутствие доступа к SSL-сертификатам — архитектурное ограничение Chrome Extension API. Однако фишинговые сайты часто используют валидные сертификаты (Let's Encrypt выдаёт их бесплатно), поэтому даже с доступом к SSL эта проверка не была бы надёжной. Наши три метрики анализируют более существенные признаки.»_

### 3. Возможность обхода алгоритма злоумышленником

| Вектор атаки | Описание | Сложность для атакующего | Митигация |
|---|---|---|---|
| Рендомизация CSS-классов | Генерация уникальных имён классов | Средняя | Использование стабильных селекторов (form, input) |
| Подмена текста картинками | Брендовые слова — не текст, а изображения | Высокая | OCR (в планах развития) |
| Unicode-подмена | `ѕberbank.ru` (кириллическая «ѕ») | Средняя | Punycode-нормализация (в планах) |
| iframe-внедрение | Фишинговая форма внутри iframe | Низкая | Проверка iframe в content.js |
| Генерация текста скриптом | Текст появляется после выполнения JS | Средняя | MutationObserver отслеживает изменения |
| Отсутствие брендовых слов | Фишинговый сайт не использует название бренда | Низкая | Снижает score, но домен может остаться подозрительным |

**Общая оценка**: Алгоритм уязвим для целенаправленного обхода, но это характерно для всех rule-based систем. Для школьного проекта это допустимо.

**Что сказать на защите**:
_«Любая система обнаружения фишинга может быть обойдена целенаправленным злоумышленником — это фундаментальное свойство задач информационной безопасности. Наша цель — не абсолютная защита, а существенное повышение порога атаки. Три метрики требуют одновременного обхода трёх разных механизмов, что значительно усложняет задачу атакующего.»_

### 4. Влияние изменений фронтенда сайтов

**Проблема**: Легитимные сайты регулярно обновляют вёрстку:
- Сбербанк меняет CSS-классы каждые 2-4 недели
- ВКонтакте перешёл на React — полная переработка DOM
- Яндекс обновляет форму авторизации

**Влияние**: Селекторы в rules.json устаревают → FN или FP.

**Митигация**:
1. Использование **стабильных селекторов** (input[type='password'], form[action*='login']) вместо специфичных CSS-классов
2. **Версионирование** rules.json — при обновлении селекторов увеличивается patch-версия
3. **Регулярное обновление** базы (в плане развития — автоматизация)

**Что сказать на защите**:
_«Устаревание селекторов — реальная проблема, характерная для всех систем, зависящих от DOM. Мы решаем её через: (1) использование максимально стабильных селекторов, (2) версионирование базы правил, (3) возможность обновления базы без изменения кода расширения.»_

### 5. Ограничение количества сервисов

**Проблема**: База правил содержит 5 сервисов. Фишинг может имитировать:
- Другие банки (Альфа-Банк, ВТБ, Райффайзенбанк)
- Почтовые сервисы (Mail.ru, Яндекс.Почта)
- Маркетплейсы (Wildberries, Ozon)
- Мессенджеры (Telegram Web)

**Влияние**: Фишинг неизвестных сервисов не обнаруживается (FN).

**Митигация**: Архитектура позволяет добавлять сервисы — достаточно добавить запись в rules.json.

### 6. Ограничения по языку

**Проблема**: Ключевые слова — только на русском. Анлоязычный фишинг (PayPal, Amazon) не обнаруживается.

**Митигация**: Добавление англоязычных keyword'ов — простое расширение базы правил.

### 1. Интеграция с PhishTank

**PhishTank** — общедоступная база данных фишинговых URL (https://phishtank.org).

**Как интегрировать**:
```javascript
// Будущая реализация
async function checkPhishTank(url) {
  // PhishTank API: https://checkurl.phishtank.com/checkurl/
  const response = await fetch('https://checkurl.phishtank.com/checkurl/', {
    method: 'POST',
    headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
    body: `url=${encodeURIComponent(url)}&format=json&app_key=YOUR_KEY`
  });
  
  const data = await response.json();
  return data.results.in_database; // true если URL в базе фишинга
}
```

**Почему это важно**:
- Актуальная база фишинговых URL, обновляемая сообществом
- Дополнительный источник данных, независимый от нашей модели
- Повышает Recall

**Ограничения**:
- Требует интернет-соединение
- Зависит от доступности API
- Есть лимит запросов

### 2. Регулярное обновление базы правил

**Подход**:
- Создать скрипт для автоматического обновления rules.json
- Использовать GitHub Actions для еженедельной проверки селекторов
- Версионирование через Git

```yaml
# .github/workflows/update-rules.yml (будущая реализация)
name: Update Rules
on:
  schedule:
    - cron: '0 0 * * 0'  # Каждое воскресенье
jobs:
  update:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Check selectors
        run: node scripts/verify-selectors.js
      - name: Update rules.json
        run: node scripts/update-rules.js
      - name: Commit if changed
        run: |
          git config user.name "Bot"
          git diff --quiet rules.json || git commit -am "Update rules"
          git push
```

### 4. Улучшение скоринга

**Возможные улучшения**:

1. **Нелинейная модель**: вместо простых весов использовать логистическую функцию:
   ```
   P(phishing) = 1 / (1 + e^(-z))
   где z = w₁×m₁ + w₂×m₂ + w₃×m₃ + b
   ```

2. **Нечёткие веса**: вместо 0/1 использовать непрерывные значения:
   ```
   m_selector = matched_count / total_selectors  // от 0 до 1
   ```

3. **Контекстные веса**: учитывать тип страницы:
   - Страница входа → домен важнее
   - Страница оплаты → селекторы важнее

### 6. Punycode-защита

**Проблема**: домены вида `xn--sberbank-...` визуально отображаются как `sberbank.ru`.

**Решение**:
```javascript
function normalizeDomain(domain) {
  // Преобразуем punycode в Unicode
  try {
    const unicode = new URL('https://' + domain).hostname;
    return unicode.toLowerCase();
  } catch {
    return domain.toLowerCase();
  }
}
```
