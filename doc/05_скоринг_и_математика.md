# PhishBuster RU — Алгоритм скоринга и математика

## 1. Весовая модель

### Общая формула

Итоговый score вычисляется как **взвешенная сумма** отдельных метрик:

```
Score = Σ (wᵢ × mᵢ) + bonus

Где:
  wᵢ — вес i-й метрики
  mᵢ — нормализованное значение i-й метрики (0 или 1)
  bonus — дополнительные баллы за urgency-сигналы
```

Развёрнутая формула для трёх основных метрик:

```
Score = w_domain × m_domain + w_selector × m_selector + w_keyword × m_keyword + w_urgency × m_urgency

Где:
  w_domain   = 40   (вес доменной метрики)
  w_selector = 35   (вес селекторной метрики)
  w_keyword  = 25   (вес ключевых слов)
  w_urgency  = 15   (вес urgency-сигналов — бонус)
```

### Нормализация

Каждая метрика `mᵢ` принимает значение **0 или 1**:

```
m_domain   = 1, если домен подозрительный;  0, если легитимный или нейтральный
m_selector = 1, если найдены характерные селекторы;  0, если не найдены
m_keyword  = 1, если найдены ключевые слова в title/body;  0, если не найдены
m_urgency  = 1, если найдены urgency-слова;  0, если не найдены
```

**Итоговая формула**:

```
Score = 40 × m_domain + 35 × m_selector + 25 × m_keyword + 15 × m_urgency
```

```
Score_final = min(Score, 100)
```

## 2. Обоснование выбора весов

### Принцип распределения

Веса распределены по принципу **достоверности признака**:

| Приоритет | Метрика | Вес | Достоверность |
|---|---|---|---|
| 1 | Домен | 40 | **Максимальная** — домен нельзя подделать на легитимном сайте |
| 2 | DOM-селекторы | 35 | **Высокая** — структура HTML уникальна для каждого сервиса |
| 3 | Ключевые слова | 25 | **Средняя** — слова могут использоваться в легитимных контекстах |
| 4 | Urgency-сигналы | 15 | **Низкая** — бонусный признак, усиливает подозрение |

### Почему домен имеет наибольший вес

1. **Фишинг по определению** использует поддельный домен. Это основной критерий.
2. **Проверка детерминирована**: домен — это строка, совпадение проверяется однозначно.
3. **Не зависит от вёрстки**: даже если фишинговый сайт полностью скопировал HTML, домен выдадут его.

## 3. Пример вычисления

### Сценарий A: Фишинговый сайт Сбербанка

**Исходные данные**:
- URL: `https://sberbank-secure-login.xyz/auth`
- Title: «Сбербанк Онлайн — Вход в систему»
- DOM содержит: `<form action="/auth">`, `<input type="password">`, `<img alt="Сбер">`
- Body содержит: «срочно заблокирован», «немедленно подтвердите»

**Вычисление метрик**:

```
1. Доменная метрика:
   Домен: sberbank-secure-login.xyz
   Совпадает с suspiciousPattern "sberbank-[a-z0-9]+\\." → ДА
   Совпадает с legitimate domain "sberbank.ru" → НЕТ
   → m_domain = 1 (подозрительный)
   → Начислено: 40 × 1 = 40 баллов

2. Селекторная метрика:
   Найдены: form[action*='auth'], input[type='password'], img[alt*='Сбер']
   Количество совпадений: 3 из 9 возможных → достаточно для confident match
   → m_selector = 1 (селекторы найдены)
   → Начислено: 35 × 1 = 35 баллов

3. Метрика ключевых слов:
   Title содержит: "Сбербанк Онлайн" → совпадение с titleKeywords
   → m_keyword = 1 (ключевые слова найдены)
   → Начислено: 25 × 1 = 25 баллов

4. Urgency-сигналы:
   Body содержит: "срочно заблокирован", "немедленно подтвердите"
   → m_urgency = 1 (urgency-слова найдены)
   → Начислено: 15 × 1 = 15 баллов
```

**Итог**:
```
Score = 40 × 1 + 35 × 1 + 25 × 1 + 15 × 1 = 115
Score_final = min(115, 100) = 100
```

**Вердикт**: `danger` 🔴 — ФИШИНГ!

### Сценарий B: Легитимный сайт Сбербанка

**Исходные данные**:
- URL: `https://online.sberbank.ru/CSAFront/index.do`
- Title: «СберБанк Онлайн»
- DOM содержит стандартные селекторы Сбербанка

**Вычисление метрик**:

```
1. Доменная метрика:
   Домен: online.sberbank.ru
   Совпадает с legitimate: "online.sberbank.ru" → ДА
   → m_domain = 0 (домен ЛЕГИТИМНЫЙ — не подозрительный!)
   → Начислено: 40 × 0 = 0 баллов

2. Селекторная метрика:
   Найдены селекторы Сбербанка → m_selector = 1
   → Начислено: 35 × 1 = 35 баллов

3. Метрика ключевых слов:
   Title: "СберБанк Онлайн" → m_keyword = 1
   → Начислено: 25 × 1 = 25 баллов

4. Urgency-сигналы:
   Body НЕ содержит urgency-слов → m_urgency = 0
   → Начислено: 15 × 0 = 0 баллов
```

**Итог**:
```
Score = 0 + 35 + 25 + 0 = 60
```

**Вердикт**: `danger` (60 ≥ 60)... **ПРОБЛЕМА!**

### Важное уточнение: логика принятия решения

Вышеприведённая формула требует **критического уточнения**. Модель должна учитывать:

**Если домен ЛЕГИТИМНЫЙ → score = 0 (автоматически безопасно)**

Обновлённый алгоритм:

```
Если домен совпадает с legitimate-списком:
    → verdict = "safe", score = 0
    → Пропустить все дальнейшие проверки

Если домен совпадает с suspiciousPattern:
    → m_domain = 1

Если домен НЕ совпадает ни с legitimate, ни с suspicious:
    → m_domain = 0 (нейтральный домен — может быть любым сайтом)
```

**Исправленный алгоритм для легитимного сайта**:
```
Домен online.sberbank.ru ∈ legitimate →
  → verdict = "safe" (досрочно)
  → score = 0
```

**Исправленный алгоритм для фишингового сайта**:
```
Домен sberbank-secure-login.xyz ∉ legitimate →
  Домен совпадает с suspiciousPattern → m_domain = 1
  Селекторы найдены → m_selector = 1
  Ключевые слова найдены → m_keyword = 1
  Urgency найдены → m_urgency = 1
  Score = 100 → verdict = "danger"
```

## 4. Влияние изменения threshold

| Threshold | Эффект | Рекомендация |
|---|---|---|
| 40 | Больше False Positives (FP↑), меньше False Negatives (FN↓) | Слишком чувствительно |
| 50 | Баланс, но возможны FP | Допустимо для бета-версии |
| **60** | **Оптимальный баланс** — нужно ≥2 совпадения метрик | **Рекомендовано** |
| 70 | Меньше FP, но больше FN (пропускает фишинг) | Слишком строго |
| 80 | Почти нет FP, но много FN | Только для критичных систем |

### Графическое представление

```
FP Rate
  ^
  │\
  │ \
  │  \     ← Чем ниже threshold, тем больше FP
  │   \
  │────\──────────── Threshold
  │     \
  │      \   ← Чем выше threshold, тем больше FN
  │       \
  └──────────────→ FN Rate
  
  Оптимальная точка: threshold ≈ 60
```

## 5. Вычислительная сложность

### Анализ сложности алгоритма скоринга

Пусть:
- **S** — количество сервисов в базе (S = 5)
- **P** — количество suspicious patterns на сервис (P ≈ 5)
- **K** — количество ключевых слов на сервис (K ≈ 15)
- **C** — количество CSS-селекторов на сервис (C ≈ 10)

**Шаг 1: Поиск совпадающего сервиса**
- Проверка домена: O(S × P) — для каждого сервиса проверяем каждый pattern
- Проверка ключевых слов: O(S × K) — для каждого сервиса проверяем каждое слово
- Итого: **O(S × max(P, K)) = O(5 × 15) = O(75) — константа**

**Шаг 2: Проверка селекторов**
- `document.querySelectorAll()` — O(E), где E — количество элементов на странице
- Практически: **O(C × E), но C ≈ 10, E ≈ 1000 → O(10 000)**

**Шаг 3: Вычисление score**
- **O(1)** — простое сложение

**Итого**: **O(S × K + C × E) ≈ O(75 + 10 000) ≈ O(10 000)**

Для S = 5 и типичной веб-страницы: **< 1 мс** на современном компьютере.

### Масштабируемость

| Параметр | Текущее значение | При увеличении |
|---|---|---|
| Количество сервисов | 5 | Линейный рост: O(S) |
| Количество паттернов | 5/сервис | Линейный рост: O(S×P) |
| Количество элементов DOM | ~1000 | Определяется сайтом, не нами |

**Вывод**: алгоритм легко масштабируется до 50-100 сервисов без заметного замедления.

## 6. Псевдокод полного алгоритма скоринга

```
ФУНКЦИЯ analyzePage(metrics, rulesDatabase):
  
  // Шаг 1: Проверка на легитимный домен
  ДЛЯ КАЖДОГО service В rulesDatabase.services:
    ЕСЛИ metrics.domain ∈ service.domains.legitimate:
      ВЕРНУТЬ { score: 0, verdict: "safe", reasons: ["Легитимный домен"] }
  
  // Шаг 2: Определение целевого сервиса
  matchedService = NULL
  maxConfidence = 0
  
  ДЛЯ КАЖДОГО service В rulesDatabase.services:
    confidence = 0
    
    // Проверка ключевых слов в title
    ДЛЯ КАЖДОГО keyword В service.keywords.titleKeywords:
      ЕСЛИ keyword ∈ metrics.title:
        confidence += 0.4
    
    // Проверка ключевых слов в body
    ДЛЯ КАЖДОГО keyword В service.keywords.bodyKeywords:
      ЕСЛИ keyword ∈ metrics.bodyText:
        confidence += 0.2
    
    ЕСЛИ confidence > maxConfidence:
      maxConfidence = confidence
      matchedService = service
  
  // Шаг 3: Если ни один сервис не совпал
  ЕСЛИ maxConfidence < 0.3:
    ВЕРНУТЬ { score: 0, verdict: "neutral", reasons: ["Не похож на известный сервис"] }
  
  // Шаг 4: Вычисление score
  score = 0
  reasons = []
  
  // 4a: Доменная метрика
  isSuspiciousDomain = ЛОЖЬ
  ДЛЯ КАЖДОГО pattern В matchedService.domains.suspiciousPatterns:
    ЕСЛИ regexMatch(metrics.domain, pattern):
      isSuspiciousDomain = ИСТИНА
  
  ЕСЛИ isSuspiciousDomain:
    score += 40
    reasons.append("Подозрительный домен")
  
  // 4b: Селекторная метрика
  matchedSelectors = countMatchedSelectors(metrics.selectors, matchedService.selectors)
  ЕСЛИ matchedSelectors >= 2:
    score += 35
    reasons.append("Найдены фирменные элементы интерфейса")
  
  // 4c: Ключевые слова
  keywordMatches = countKeywordMatches(metrics.title, metrics.bodyText, matchedService.keywords)
  ЕСЛИ keywordMatches >= 1:
    score += 25
    reasons.append("Совпадение ключевых слов")
  
  // 4d: Urgency-сигналы
  urgencyMatches = countUrgencyMatches(metrics.bodyText, matchedService.keywords.urgencyKeywords)
  ЕСЛИ urgencyMatches >= 1:
    score += 15
    reasons.append("Обнаружены признаки социальной инженерии")
  
  // Шаг 5: Определение вердикта
  score = min(score, 100)
  
  ЕСЛИ score >= 60: verdict = "danger"
  ИНАЧЕ ЕСЛИ score >= 30: verdict = "warning"  
  ИНАЧЕ: verdict = "safe"
  
  ВЕРНУТЬ { score, verdict, reasons, serviceName: matchedService.name }
КОНЕЦ ФУНКЦИИ
```
